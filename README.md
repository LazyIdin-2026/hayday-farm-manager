# hayday-app — dashboard (React) + storage + หน้า Inventory + พร้อม deploy ขึ้น GitHub Pages

## 🚀 วิธี deploy ขึ้น GitHub Pages (อ่านก่อนใช้งานจริง)

**ทำไมผมส่งเป็น .zip ให้ทำเองแทนที่จะ push ให้เลย:** แซนด์บ็อกซ์ที่ผมรันอยู่บล็อกการเชื่อมต่อไปยัง github.com/git โดยตรง (แม้แต่ `git push`/`api.github.com` เกือบทุก endpoint ก็ถูกบล็อก ตรวจสอบแล้วจริงๆ ไม่ใช่แค่เดา) เลยต้อง push เองจากฝั่งคุณ — ข่าวดีคือทำได้หมดจากแท็บเล็ตเลย ไม่ต้องมีคอมพิวเตอร์ ผ่าน **GitHub Codespaces** (VS Code บนเว็บ ฟรีสำหรับ repo ส่วนตัว/สาธารณะในโควตามาตรฐาน):

1. เปิด github.com (แอพหรือเบราว์เซอร์บนแท็บเล็ตก็ได้) → สร้าง repo ใหม่ชื่อ **`hayday-farm-manager`** ตั้งเป็น **Public** ไม่ต้องติ๊ก README/gitignore (ปล่อยว่างไว้)
2. ในหน้า repo ที่เพิ่งสร้าง กดปุ่มเขียว **`< > Code`** → แท็บ **Codespaces** → **Create codespace on main**
3. รอ VS Code บนเว็บโหลดเสร็จ (ครั้งแรกอาจใช้เวลา ~1 นาที)
4. ใน panel **Explorer** ทางซ้าย คลิกขวาที่พื้นที่ว่าง → **Upload...** → เลือกไฟล์ `hayday-app.zip` ที่ผมส่งให้ (อัปโหลดขึ้นไปที่ root ของ Codespace)
5. เปิด **Terminal** (เมนู ☰ → Terminal → New Terminal) แล้วรันทีละบรรทัด:
   ```bash
   unzip -o hayday-app.zip -d /tmp/extracted
   cp -r /tmp/extracted/hayday-app/. .
   rm -rf /tmp/extracted hayday-app.zip
   git add -A
   git commit -m "Initial commit: Hay Day Multi-Account Farm Manager"
   git push
   ```
6. ไปที่แท็บ **Settings** ของ repo → เมนูซ้าย **Pages** → หัวข้อ "Build and deployment" → Source เลือกเป็น **GitHub Actions** (ทำครั้งเดียว ถ้า workflow ตรวจจับอัตโนมัติแล้วอาจข้ามขั้นตอนนี้ได้เลย)
7. ไปที่แท็บ **Actions** ของ repo รอ workflow "Deploy to GitHub Pages" รันจนขึ้นติ๊กเขียว (ประมาณ 1-2 นาที — ขั้นตอนนี้จะ `npm install` + รันเทสทั้ง 51 เคสอัตโนมัติ + build จริงด้วย Vite แล้วค่อย deploy ถ้าเทสพังจะไม่ deploy ให้ ป้องกันโค้ดพังหลุดขึ้นเว็บจริง)
8. เว็บจะขึ้นที่ **`https://<username>.github.io/hayday-farm-manager/`** (แทน `<username>` ด้วยชื่อบัญชี GitHub ของคุณ) เปิดลิงก์นี้บน Xiaomi Pad ได้เลย ถ้าอยากได้ความรู้สึกเหมือนแอพ กด "เพิ่มลงหน้าจอหลัก" จากเมนูเบราว์เซอร์ได้ด้วย

ถ้ามีคอมพิวเตอร์ใช้ตอนไหนก็สลับไปใช้วิธีมาตรฐานแทนได้: `git clone` repo ว่างที่สร้างไว้ → แตกไฟล์ zip นี้ทับ → `git add -A && git commit -m "..." && git push`

**สิ่งที่ผมตรวจสอบแล้วจริงก่อนส่ง:** เทสทั้งหมด 51 เคสผ่าน + รันแอพจริงใน Chromium แบบ headless (ไม่ใช่แค่ SSR) ยืนยันว่า mount ได้ไม่มี error, กดปุ่ม "เพิ่มบัญชีฟาร์ม" ผ่าน modal จริงแล้วข้อมูลบันทึกลง IndexedDB จริงและขึ้นแดชบอร์ดจริง, สลับหน้า/สลับโหมด "ดูทุกบัญชีพร้อมกัน" ในหน้าคลังสินค้าทำงานถูกต้อง — **ส่วนที่ตรวจสอบในแซนด์บ็อกซ์นี้ไม่ได้คือหน้าตา Tailwind CSS จริง** (แซนด์บ็อกซ์บล็อกการติดตั้ง Tailwind/เชื่อมต่อ CDN ออกไปนอกเครื่อง) แต่ตั้งค่า Tailwind ผ่าน Vite มาตรฐานเป๊ะ (`tailwind.config.js`+`postcss.config.js`) ซึ่งเป็นวิธีตั้งค่าที่ใช้กันแพร่หลายมาก มั่นใจว่าจะแสดงผลถูกต้องตอน build จริงบน GitHub Actions (ที่ต่ออินเทอร์เน็ตได้เต็มที่) — เช็คหน้าตาอีกทีหลัง deploy เสร็จได้เลย ถ้ามีจุดไหนเพี้ยนบอกมาได้ แก้ให้ต่อได้ทันที


ผมรวม `farm-dashboard.jsx` ที่คุณแนบมา เข้ากับ data layer (`hayday-storage`) เข้าด้วยกัน ต่อ modal form สวยๆ แทน `prompt()`, แยกสีครบ 6 ประเภทกิจกรรมจริง, เพิ่ม **หน้าคลังสินค้าแยกต่างหาก** (เฟส 4), เพิ่ม **ปุ่มสลับมุมมองในหน้า Inventory** (เฟส 5) และล่าสุด (เฟส 6) เพิ่ม **แก้ไข/ลบ/เก็บกิจกรรมได้จากแดชบอร์ด** — แตะรายการกิจกรรมเพื่อแก้จำนวน/เวลาที่เหลือ, ลบทิ้ง, หรือเก็บ/ทำเครื่องหมายว่าส่งแล้ว — **รันจริงในแซนด์บ็อกซ์ของผมเองจนผ่านหมดแล้ว**

## เฟส 6: แก้ไข/ลบ/เก็บกิจกรรมจากแดชบอร์ด

ที่มา: ตอนดูเว็บจริงบนแท็บเล็ต เจอกิจกรรม "ข้าวสาลี x76 — 3703ชม 48น" ที่กรอกเวลาผิดหน่วยตอนเพิ่ม (พิมพ์เป็นนาทีแต่ระบบตีความเป็นชั่วโมง) แต่ตอนนั้นแดชบอร์ดยังไม่มีทางแก้ไข ได้แต่ลบบัญชีทั้งบัญชีแล้วสร้างใหม่ — เลยเพิ่มฟีเจอร์แก้ไข/ลบ/เก็บกิจกรรมแบบแตะแล้วเปิด modal ได้เลย ไม่ต้องลบทั้งบัญชี

**ใช้งานยังไง:** แตะที่การ์ดกิจกรรมใบไหนก็ได้ในแดชบอร์ด (การ์ดตอบสนองการแตะแล้ว — ไม่ใช่แค่ปุ่ม "เพิ่มกิจกรรม" อีกต่อไป) จะเปิด modal ให้:
- **พืชผล/สัตว์เลี้ยง/โรงงาน/โรงงานอีเวนต์** — แก้ "จำนวน" และ "เวลาที่เหลือ" ได้ตรงๆ (เลือกหน่วยนาที/ชั่วโมงได้) กด "บันทึก" อัปเดตทันที
- **ออเดอร์เรือ/รถ, ออเดอร์เมือง** — ยังแก้รายการไอเทมจากหน้านี้ไม่ได้ (ดูข้อจำกัดด้านล่าง) แต่ลบทิ้งได้ หรือกด "ทำเครื่องหมายว่าส่งแล้ว" ถ้าครบตามที่ต้องส่งแล้ว
- ทุกประเภท กด **"ลบ"** ได้ (กดครั้งแรกเปลี่ยนเป็น "ยืนยันลบ?" กันกดพลาด กดอีกครั้งถึงลบจริง)
- ถ้ากิจกรรม/ออเดอร์นั้น "พร้อมเก็บแล้ว" จะมีปุ่มลัดสีชมพู-ม่วงเด่นด้านบน modal ให้เก็บ/ทำเครื่องหมายว่าส่งแล้วได้ในคลิกเดียว โดยไม่ต้องกดบันทึกฟอร์มก่อน

ไฟล์ที่เปลี่ยน/เพิ่ม:
- `storage/repository.js` (+ sync ไปที่ `hayday-storage/src/repository.js`) — เพิ่ม `updateActivity`/`deleteActivity`/`updateEventProduction`/`deleteEventProduction`/`deleteBoatTruckOrder`/`deleteTownOrder` แก้/ลบ record ตรงๆ ผ่าน adapter เดิม (ออเดอร์ลบรายการไอเทมลูกของตัวเองไปด้วยอัตโนมัติ — ระวังจุดที่ 2 ตารางนี้ใช้ชื่อ index ไอเทมลูกต่างกัน: `boatTruckOrderItems` ใช้ `orderId`, `townOrderItems` ใช้ `townOrderId`)
- `src/mappers.js` — แต่ละ item ที่ `mapAccountsForDashboard` คืนออกมา เพิ่มฟิลด์ `sourceId` (ไอดี record จริงในตารางต้นทาง) และ `quantity` (ตัวเลขดิบ) ไว้ใช้ dispatch การแก้ไข/ลบกลับไปที่ record ที่ถูกต้อง
- `src/formkit.jsx` — เพิ่ม `DangerButton` (ปุ่มแดง-ชมพูไล่เฉด) ใช้กับปุ่มลบ
- `src/EditActivityModal.jsx` (ใหม่) — modal เดียวรองรับทั้ง 6 ประเภท: กิจกรรมเดี่ยว 4 ประเภทแก้ไขได้เต็มที่, ออเดอร์ 2 ประเภทลบ/ทำเครื่องหมายว่าส่งแล้วได้ (ยังแก้ไขรายการไอเทมไม่ได้)
- `src/actions.js` — เพิ่ม `updateActivityItemAction`/`deleteActivityItemAction`/`collectActivityItemAction`/`fulfillOrderItemAction` แต่ละตัว dispatch ไปที่ repo function ที่ถูกต้องตาม `item.type` โดยใช้ `item.sourceId`
- `src/FarmDashboard.jsx` — `ItemRow` แตะได้แล้ว (`role="button"`, คีย์บอร์ด Enter/Space ใช้ได้ด้วย), ส่ง `onEditActivity` ผ่าน `AccountCard` ลงไปจนถึง `ItemRow`
- `src/App.jsx` — เพิ่ม state `editActivityItem` + handler ทั้ง 4 (`handleOpenEditActivity`, `handleSaveActivityItem`, `handleDeleteActivityItem`, `handleCollectActivityItem`, `handleFulfillOrderItem`) เรียก action แล้ว `reload()` ทุกครั้งให้แดชบอร์ด/คลังสินค้าสดใหม่เสมอ (error จาก repo เช่นสต็อกไม่พอตอนทำเครื่องหมายว่าส่งออเดอร์ จะโชว์เป็น `window.alert` ไม่ทำให้แอพล่ม)

เทสใหม่รอบนี้ (7 เคส ใน `test/actions.test.js`):
- `updateActivityItemAction` แก้จำนวน/เวลาของกิจกรรมเดี่ยวได้ถูกต้อง (รวมเคสจำลองบั๊กจริงที่เจอ — สร้างกิจกรรม 76 หน่วย/3703 ชม. แล้วแก้ให้เหลือเวลาที่ถูกต้อง ~55 นาที ยืนยันว่าแก้ได้จริง) และ throw ถ้าเป็นประเภทออเดอร์
- `deleteActivityItemAction` ลบได้ถูกต้องครบทั้ง 6 ประเภทในเทสเดียว
- `collectActivityItemAction` เก็บกิจกรรมแล้วสต็อกเพิ่มจริง + รายการหายจาก feed
- `fulfillOrderItemAction` ทำเครื่องหมายว่าส่งออเดอร์สำเร็จเมื่อสต็อกพอ และ throw เมื่อสต็อกไม่พอ

**ข้อจำกัดที่ตั้งใจเว้นไว้**: หน้านี้แก้ได้แค่ "จำนวน" กับ "เวลาที่เหลือ" ของกิจกรรมเดี่ยว — เปลี่ยนชนิดไอเทม/ชื่อกิจกรรมยังทำไม่ได้ (ลบแล้วสร้างใหม่แทน) และออเดอร์ (เรือ/รถ/เมือง) ยังแก้รายการไอเทมที่ต้องส่งจากหน้านี้ไม่ได้เลย ทำได้แค่ลบทิ้งหรือทำเครื่องหมายว่าส่งแล้ว — ถ้าอยากได้ฟีเจอร์แก้ไขรายการไอเทมในออเดอร์ บอกได้ เพิ่มได้ไม่ยาก

## เฟส 5: ปุ่มสลับมุมมองในหน้า Inventory

หน้า Inventory ตอนนี้มีปุ่มสลับมุมมอง 2 โหมด (สไตล์ pill เข้าชุดกับ `NavTabs` เดิม) วางไว้ใต้หัวข้อ "คลังสินค้า" สลับไปมาได้ในหน้าเดียวโดยไม่ต้องออกจากหน้า:

- **"ดูทีละบัญชี"** — พฤติกรรมเดิมทุกอย่างจากเฟส 4 (pill เลือกบัญชี + ตารางไอเทมของบัญชีที่เลือก แก้ไขจำนวน/เกณฑ์แบบ inline)
- **"ดูทุกบัญชีพร้อมกัน"** — ตารางรวม **ไอเทม × บัญชี** เทียบจำนวนคงเหลือของทุกบัญชีข้ามกันได้ในตารางเดียว (คอลัมน์ = บัญชี, แถว = ไอเทมในแคตตาล็อก) ช่องที่ต่ำกว่าเกณฑ์ของบัญชีนั้นๆ ไฮไลต์สีส้ม-ชมพูพร้อมไอคอนเตือนเฉพาะ cell ที่ต่ำ (เกณฑ์ของแต่ละบัญชีแยกกันอิสระ เหมือนเดิม)

การบันทึกยังเป็น**อัตโนมัติแบบเดิมทุกอย่าง** — พิมพ์แล้วออกจากช่อง (blur) = บันทึกทันที ไม่มีปุ่มยืนยันเพิ่ม ใช้ `EditableNumberCell` ตัวเดียวกันทั้ง 2 โหมด (แยกออกมาเป็น `src/EditableNumberCell.jsx` เพื่อใช้ซ้ำ)

ไฟล์ที่เปลี่ยน/เพิ่ม:
- `src/inventoryMappers.js` — เพิ่มฟังก์ชันใหม่ `mapInventoryMatrix(catalogItems, categories, accounts, inventoryRowsByAccountId)` คืนแถวต่อไอเทม แต่ละแถวมี `cells[]` เรียงลำดับตาม `accounts` ที่ส่งเข้ามา (แต่ละ cell มี `{accountId, quantity, lowStockThreshold, isLow}`) — ไอเทม/บัญชีคู่ไหนไม่เคยตั้งสต็อกมาก่อนได้ค่า default เหมือน `mapInventoryForAccount` เดิม (quantity 0, threshold null, isLow false)
- `src/EditableNumberCell.jsx` (แยกออกมาใหม่) — ช่องตัวเลข inline-edit เดิมจาก `InventoryPage.jsx` ย้ายมาไว้ไฟล์แยกเพื่อใช้ร่วมกันทั้ง 2 ตาราง โค้ดข้างในไม่เปลี่ยนแม้แต่บรรทัดเดียว
- `src/InventoryPage.jsx` — เพิ่มปุ่มสลับโหมด (`ViewModeToggle`) + รับ prop ใหม่ `matrixItems`, `viewMode`, `onChangeViewMode` เรนเดอร์ตารางเทียบข้ามบัญชีเมื่อ `viewMode === "all"` (ตารางเดิมของโหมด "ดูทีละบัญชี" ไม่ถูกแตะเลย ยกเว้นเปลี่ยน `onUpdateQuantity(itemId, v)` → `onUpdateQuantity(selectedAccountId, itemId, v)` เพื่อให้ signature ตรงกับที่ใช้ร่วมกับตารางเทียบ)
- `src/App.jsx` — เพิ่ม state `inventoryViewMode` ('single'/'all') และ `inventoryMatrix`, ฟังก์ชันใหม่ `reloadInventoryMatrix(mappedAccounts, repo)` โหลดสต็อกของทุกบัญชีพร้อมกันแล้วแปลงผ่าน `mapInventoryMatrix` (เรียกอัตโนมัติทุกครั้งที่ `reload()` ทำงาน เพื่อให้ตารางเทียบสดใหม่เสมอไม่ว่าจะแก้จากหน้าไหน) และ**ปรับ signature ของ `handleUpdateQuantity`/`handleUpdateThreshold` ให้รับ `accountId` ตรงๆ** เป็นพารามิเตอร์แรก (ไม่ implicit ใช้บัญชีที่เลือกอยู่แบบเดิมอีกต่อไป) เพื่อให้ใช้ handler ชุดเดียวกันได้ทั้ง 2 โหมด — หลังแก้ค่าแล้วรีเฟรชทั้งแดชบอร์ด (แบนเนอร์แจ้งเตือน), ตารางเทียบ, และตารางบัญชีเดี่ยว (เฉพาะกรณีแก้บัญชีที่เลือกอยู่) ให้สอดคล้องกันเสมอ

เทสใหม่รอบนี้ (7 เคส):
- `mapInventoryMatrix` เรียง cells ตามลำดับ accounts ที่ส่งเข้ามา, คำนวณ `isLow` ถูกต้อง**ต่อบัญชี**อิสระจากกัน, ใส่ค่า default ถูกต้องเมื่อบัญชีไหนไม่เคยตั้งสต็อกไอเทมนั้น (แม้ไม่มีคีย์ใน Map เลยด้วยซ้ำ), เรียงตามหมวด+ชื่อภาษาไทยเหมือน `mapInventoryForAccount`, รับ accounts ว่างได้ไม่ throw
- `InventoryPage` แสดงปุ่มสลับมุมมองทั้ง 2 label เสมอ, เรนเดอร์ตารางเทียบข้ามบัญชีเต็มสาย storage → `mapInventoryMatrix` → component (ยืนยันชื่อบัญชีทั้งคู่ขึ้นเป็นหัวคอลัมน์ + ค่าคงเหลือของแต่ละบัญชีถูกต้อง), ไม่ throw เมื่อยังไม่มีบัญชีเลยในโหมด "all" (ตกไปที่ empty state เดียวกับโหมด single)

**ข้อจำกัดเดิมยังอยู่เหมือนเฟส 4**: ทดสอบ interaction พิมพ์+blur จริงในเบราว์เซอร์ยังทำไม่ได้ในแซนด์บ็อกซ์นี้ (ไม่มี DOM จำลอง) แต่ตรวจสอบ pipeline ข้อมูลจริงทั้งสาย (storage → mapper → component) แสดงค่าถูกต้อง 100% ผ่านการเรนเดอร์ SSR แล้ว

## จุดที่ยังไม่ได้ทำ / ควรรู้ก่อนใช้จริง

- **Modal ยังไม่ทำ focus trap เต็มรูปแบบ** — กด Tab จะยังหลุดออกนอก modal ได้ (ปิดด้วย ESC/คลิกนอกกรอบได้ปกติ) ถ้าต้องการ accessibility เต็มที่ค่อยเพิ่ม focus trap ทีหลังได้
- **วันจบอีเวนต์ตอนนี้เลือกเองได้แล้ว** (ช่อง "วันจบอีเวนต์" แบบ `type="date"` prefill เป็น 7 วันจากนี้ไว้ให้แก้ไขสะดวก แต่เปลี่ยนเป็นวันไหนก็ได้) ถ้าใช้ชื่ออีเวนต์ที่เคยสร้างไว้แล้ว ระบบจะ**คงวันจบเดิม**ไว้ไม่ทับ (กันเผลอเลื่อนวันจบอีเวนต์เดิมโดยไม่ตั้งใจตอนเพิ่มโรงงานอีเวนต์ตัวที่สอง) ถ้าอยากแก้วันจบของอีเวนต์ที่มีอยู่แล้วโดยตรง บอกได้ เพิ่มปุ่มแก้ไขอีเวนต์แยกได้ไม่ยาก
- ยังไม่ได้เชื่อมกับสูตร/วัตถุดิบจากแคตตาล็อกจริง (การกรอกชื่อกิจกรรม/ไอเทมในฟอร์มจะ auto-create แคตตาล็อกใหม่ถ้ายังไม่มี ผ่าน `findOrCreateItemByName`) — ถ้าอยากให้เลือกจากสูตรที่มีอยู่แล้ว (พร้อมตัดวัตถุดิบอัตโนมัติจริงตามสูตร) ต้องเพิ่ม dropdown เลือกสูตรในฟอร์ม ซึ่ง `repo.startActivity` รองรับอยู่แล้ว รอแค่ต่อ UI
- **หน้า Inventory (เฟส 4):** การแก้ไขจำนวน/เกณฑ์เป็นแบบ "พิมพ์แล้วออกจากช่อง (blur) = บันทึกทันที" ไม่มีปุ่ม "บันทึก" แยก — ทดสอบ interaction แบบพิมพ์-blur จริงในเบราว์เซอร์ยังทำไม่ได้ในแซนด์บ็อกซ์นี้ (ไม่มี DOM ให้จำลองการพิมพ์/blur) แต่ตรวจสอบแล้วว่า pipeline ข้อมูลจริงทั้งสาย (storage → mapper → component) แสดงค่าถูกต้อง 100%; ช่องค้นหาก็เช่นกัน — กรองผ่าน local state ในเบราว์เซอร์ ทดสอบที่ระดับข้อมูลแทน ถ้าอยากได้ปุ่ม "บันทึก" แยกจากการพิมพ์แล้วเปลี่ยนอัตโนมัติ บอกได้ ปรับได้ไม่ยาก
- **แก้ไขกิจกรรม (เฟส 6):** แก้ได้แค่ "จำนวน"/"เวลาที่เหลือ" ของกิจกรรมเดี่ยว 4 ประเภท — ออเดอร์เรือ/รถ/เมืองยังแก้รายการไอเทมที่ต้องส่งจากหน้านี้ไม่ได้ (ลบทิ้งแล้วสร้างใหม่แทนได้) และเปลี่ยนชนิดไอเทม/ชื่อกิจกรรมยังทำไม่ได้เหมือนกัน ถ้าอยากได้เพิ่ม บอกได้
